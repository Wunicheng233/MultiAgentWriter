# StoryForge AI 架构演进规划

## 1. 文档目的

这份文档不讨论 UI，也不讨论市场运营，而是专门回答一个更底层的问题：

> StoryForge AI 作为一个长期项目，应该如何把当前可运行但偏 demo 化的多智能体小说生成系统，演进成一个稳定、可扩展、可评测、可沉淀数据资产的创作基础设施？

重点覆盖四个维度：

- `多智能体创作系统` 应该如何演进，而不是继续停留在“几个 Prompt 串起来”
- `workflow` 应该如何从线性脚本演进成可恢复、可插拔、可观测的工作流系统
- `harness` 应该如何建立，才能真正持续优化生成质量，而不是靠主观试错
- `数据层` 应该如何从“文件 + 数据库 + 向量库混合状态”演进成可靠的产品底座

---

## 2. 当前架构的真实状态

结合当前仓库实现，系统实际上已经具备了一个很不错的原型骨架：

- 前端：React + Vite，具备项目创建、编辑器、阅读器、分享、导出等基础页面
- 后端：FastAPI，具备认证、项目、章节、任务、分享、导出等 API
- 异步执行：Celery + Redis
- 主存储：PostgreSQL
- 文件存储：项目目录中保存 `novel_plan.md`、`setting_bible.md`、`chapters/chapter_x.txt`
- 检索层：Chroma 向量库，用于设定与章节片段检索
- 生成核心：`core/orchestrator.py` 中的 `NovelOrchestrator`

从“能不能把小说跑出来”的角度，这套结构已经成立；但从“能不能支撑长期产品演进”的角度，它还存在几个结构性问题。

### 2.1 当前核心优点

- 有清晰的创作主路径，不是纯聊天
- 已经区分了策划、写作、评审、修订等不同角色
- 已经引入任务队列，说明系统有异步化意识
- 已经有章节确认、人机协作、分享和导出，说明产品闭环雏形存在
- 已经开始做系统级 guardrails，而不是完全依赖模型自觉

### 2.2 当前核心问题

#### A. 文档中的架构和运行中的架构已经分叉

仓库里的设计文档仍然描述更完整、更复杂的多 Agent 流程，但当前实际运行主路径已经收敛为更精简的：

`Planner -> Writer -> Guardrails -> Critic -> Revise`

这本身不是坏事，坏的是：

- 文档和真实执行不一致
- 系统后续优化缺少统一目标架构
- 很容易出现“以为系统支持，实际上代码里并没有”的长期认知偏差

长期项目里，这种分叉如果不尽快收口，会让产品设计、工程实现、评测体系三条线越来越难协同。

#### B. 工作流仍然是过程脚本，不是显式状态机

当前编排逻辑主要写在 `core/orchestrator.py`，本质上是一个线性过程：

- 读配置
- 生成 plan
- 逐章生成
- 章节内跑若干检查和修订
- 需要人工确认时抛异常暂停
- 用户确认后通过新任务重启继续

这对于 demo 很实用，但对于长期项目有几个明显限制：

- 执行状态依赖代码路径，不依赖显式工作流状态
- 恢复执行依赖文件存在与否来推断，而不是依赖标准化 run state
- 节点不可独立重放、不可单独评测、不可部分替换
- “重写当前章”“保留设定重做本章”“从第 N 章分叉”这些未来必需能力都很难优雅实现

#### C. 状态分散，缺少单一事实源

当前同一个项目的状态，可能同时存在于以下位置：

- PostgreSQL：项目、章节、任务、阅读进度、分享等
- 文件系统：`setting_bible.md`、`novel_plan.md`、`chapter_*.txt`、`feedback_*.txt`
- Celery 任务状态：进度、步骤、当前章节
- Chroma：设定与章节 embedding
- `info.json`：质量分数和过程统计

这会带来长期系统最典型的问题：

- 状态漂移
- 恢复逻辑复杂
- 调试困难
- 很难回答“现在系统认定的最新版本到底是哪一个”

#### D. Agent 输入输出契约还不够强

当前很多步骤依赖：

- Prompt 输出约定
- 文本格式解析
- 默认文件命名约定
- 运行时隐式上下文

这意味着系统升级成本会很高：

- 换模型容易破格式
- 多加一个 Agent，耦合就更高
- 评测时难以比较不同节点的真实贡献

长期来看，Agent 不是越多越好，`Agent contract` 清晰才最重要。

#### E. 人在环路还是“文件反馈驱动”，不是一等工作流事件

当前章节不通过时，系统把反馈写入 `feedback_x.txt`，然后通过新 Celery 任务继续运行。

这个方案在 demo 阶段非常实用，但不适合长期使用，因为：

- 用户反馈不是结构化数据
- 很难做反馈分析、归因和产品优化
- 无法自然支持多轮反馈历史
- 无法支持多角色协作反馈
- 无法支持未来的“读者评论反哺创作”

#### F. Harness 几乎还不存在

当前系统有评分和 guardrails，但还没有真正意义上的 `evaluation harness`：

- 没有固定评测样本集
- 没有版本化 prompt / model / 参数组合
- 没有自动回归评测
- 没有离线评测与线上指标联动
- 没有一套可以回答“为什么这次比上周更好”的证据系统

对于长期 AI 产品，这一点会比“再加两个 Agent”重要得多。

#### G. 检索层存在全局状态风险

当前向量检索依赖全局 `CURRENT_OUTPUT_DIR` 和按当前项目动态命名 collection 的方式运行。这个方案对单项目串行 demo 足够，但在长期演进中存在几个风险：

- 并发项目下的上下文隔离依赖全局变量，天然脆弱
- worker 重试、任务并行、多人协作时容易出现意外串项目
- 检索层和业务主存储耦合过紧，难以独立重建和调试

## 3. 长期架构目标

我们不应该把 StoryForge AI 的目标理解成“一个更复杂的 orchestrator”，而应该理解成：

**一个面向长篇创作的 AI 原生创作操作系统。**

它的目标不是只输出一段文本，而是稳定地产出一组可追踪、可修订、可评测、可发布的创作工件。

### 3.1 目标能力

目标架构需要支持以下能力：

1. `显式工作流`
   每一步都知道自己在什么状态、依赖什么输入、产生什么输出。

2. `工件化`
   计划、大纲、设定、章节草稿、评审意见、用户反馈、修订稿、评分结果，都应是标准化工件。

3. `可恢复`
   任意任务中断后，可以从最近稳定节点恢复，而不是靠猜文件是否存在。

4. `可分叉`
   同一部小说可以在第 N 章开始产生不同分支，用于重写、AB 实验或用户手动分线。

5. `可评测`
   每一种 Prompt、模型、工作流配置都能通过 harness 比较好坏。

6. `可解释`
   系统要能回答：为什么这一章被打回、为什么这次得分低、为什么建议这样修。

7. `可沉淀`
   创作过程中产生的数据，不只是“过程垃圾”，而是未来产品能力与社区能力的资产。

8. `风格可控且可审计`
   用户可以选择或构建“受某种叙事风格启发”的创作方向，但系统必须能记录风格来源、注入强度、披露要求和近重复风险。

## 4. 总体演进原则

### 4.1 原则一：不要从“更多 Agent”开始，要从“更强合同”开始

当前最需要的不是把 Guardian、Compliance、Polish、世界观管理器一次性全部塞回主路径，而是先把每个节点的：

- 输入格式
- 输出格式
- 成功条件
- 失败类型
- 回退策略

定义清楚。

没有契约的多 Agent，最终只会变成更贵、更慢、更难调试的单 Agent。

### 4.2 原则二：数据库是事实源，文件系统是派生缓存与导出层

长期来看，系统应该明确：

- `数据库` 保存真实业务状态
- `文件系统` 用于导出、缓存、兼容调试和本地工件快照
- `向量库` 是可重建索引，不是事实源
- `Celery state` 只是执行期元数据，不是业务真相

如果这条原则不立住，未来一做协作、多分支、发布、评论反哺，就会越来越混乱。

### 4.3 原则三：工作流引擎优先于 prompt 微调

长期收益排序应当是：

1. 状态与流程可控
2. 评测体系可用
3. 检索与上下文组织更稳定
4. 再去做 prompt 精修和 agent 扩充

否则你会不断微调 Prompt，但系统整体体验并不稳定。

### 4.4 原则四：先做单机模块化单体，不急着做微服务

StoryForge AI 未来完全可能成长为多服务系统，但现在不应该过早微服务化。

现阶段更合适的做法是：

- 保持单体仓库
- 在进程内把边界划清
- 用清晰的模块、接口和异步任务边界做解耦

先把系统做成“结构清晰的模块化单体”，而不是“分布式 demo”。

### 4.5 原则五：风格不是 Prompt 文案，而是可治理的创作策略

新调研报告提出的“女娲 skill / Hermes Agent”方向很有启发，但当前不应把项目直接重构成外部 Agent 平台。

更稳妥的路线是：

- 先把女娲式方法论吸收为内部 `Style Runtime`
- 把“作家风格”表达为结构化 `StyleProfile`，而不是散落在 prompt 里的描述
- Writer / Revise 可以强风格，Planner 只做弱风格，Critic 应承担风格审计而不是模仿作者
- Hermes 类外层自治运行时只作为二期研究线，用于技能治理、跨项目记忆和批量评测，不进入当前主路径

## 5. 目标逻辑架构

长期建议把系统拆成六个基础层，并保留一个横切的风格治理能力层。

### 5.1 Product API Layer

负责：

- 项目管理
- 章节编辑
- 发布与分享
- 评论、收藏、追更、打赏
- 权限与协作

这层不关心“Prompt 怎么写”，只关心业务动作与用户权限。

### 5.2 Creation Control Plane

这是创作域的“控制中台”，负责：

- 创建创作 run
- 启动 / 暂停 / 恢复 / 取消工作流
- 记录当前版本、分支、工件关系
- 接收用户反馈并转成结构化命令

可以把它理解为“创作任务的操作系统内核”。

### 5.3 Workflow Runtime Layer

负责真正执行创作图谱，而不是执行单个 API 请求。

长期它应该支持：

- 显式节点定义
- 节点重试
- 节点跳过
- 节点回放
- 人工审批节点
- 分支节点
- 条件路由
- 失败恢复

这一层是未来替代当前线性 `NovelOrchestrator` 的核心。

### 5.4 Agent Runtime Layer

负责封装模型调用与 Agent 能力。

每个 Agent 应该具备统一契约：

- `input schema`
- `output schema`
- `prompt version`
- `model version`
- `token usage`
- `latency`
- `error type`

只有把 Agent 变成标准化执行单元，才有可能做稳定优化。

### 5.5 Memory & Retrieval Layer

负责上下文构建，而不是只做向量搜索。

长期应该包含三类记忆：

- `Canonical Memory`
  设定圣经、角色卡、世界规则、章节大纲、时间线、伏笔清单

- `Narrative Memory`
  已确认章节、场景摘要、冲突推进、角色关系变化

- `Working Memory`
  当前节点临时上下文、用户本轮反馈、最近改动摘要

这三类记忆不能全部混在一个向量库查询里。

### 5.6 Evaluation & Observability Layer

负责：

- 离线评测
- 线上指标
- A/B 对比
- 失败分析
- 成本监控
- 版本回归

这是长期优化的发动机。

### 5.7 Style Runtime & Governance Layer

这是横切在 Agent Runtime、Memory & Retrieval、Evaluation & Observability 之上的能力层，不应被做成简单的“文风 prompt 模板”。

它长期负责：

- `StyleProfile`
  结构化描述目标风格的叙事节奏、意象偏好、表达禁区、认知框架和披露模板。

- `StyleEvidenceBundle`
  记录风格依据、公共领域状态、语料来源、译本/整理版授权风险。

- `StyleRenderContext`
  按不同 Agent 职责渲染不同强度的风格上下文，避免所有 Agent 都用同一种口吻。

- `StyleAuditResult`
  记录风格保真度、原创性风险、近重复风险、披露合规状态。

第一版落地边界：

- 不做“作者克隆”宣传，只做“受某风格启发”的创作辅助
- 只优先支持历史作者或抽象风格标签
- 不引入 Hermes 作为首期运行时
- 风格能力必须进入 harness，而不是只靠主观观感判断

## 6. Workflow 演进规划

### 6.1 当前工作流的问题本质

当前工作流虽然能跑通，但它的状态机是“藏在代码里的”。长期应该把工作流从：

`过程脚本`

演进为：

`基于节点和状态迁移的创作图`

### 6.2 建议的标准工作流对象

建议引入三个核心对象：

- `WorkflowDefinition`
  描述一条创作链路包含哪些节点、什么顺序、什么条件跳转。

- `WorkflowRun`
  某个项目、某个分支、某次生成任务的执行实例。

- `WorkflowStepRun`
  某个节点在某个 run 中的一次具体执行记录。

### 6.3 建议的标准节点类型

建议后续所有创作节点都收敛到有限几类：

- `generator`
  生成内容，如 planner / writer / scene writer

- `analyzer`
  分析内容，如 critic / consistency check / pacing check

- `transformer`
  变换内容，如 revise / summarize / structure extract

- `guardrail`
  规则检查，如标题、字数、格式、敏感内容

- `approval`
  人工审批节点

- `router`
  根据结果决定下一步路径

- `persister`
  将内容、摘要、评分、事件持久化

### 6.4 第一版目标工作流

不建议一上来就做场景级 DAG。第一版工作流引擎应该只覆盖章节级创作：

`Plan -> Plan Approval -> Chapter Draft -> Guardrails -> Critic -> Revise(if needed) -> Score -> User Approval -> Persist -> Next Chapter`

但这次要把它改造成显式状态。

建议章节状态至少包含：

- `pending`
- `context_ready`
- `draft_generated`
- `guardrails_checked`
- `critic_failed`
- `revised`
- `awaiting_user_review`
- `approved`
- `rejected`
- `persisted`
- `published`

### 6.5 第二版目标工作流

当章节级工作流稳定后，再升级为场景级或段落级：

- 章节规划
- 场景拆解
- 场景逐个生成
- 场景拼接
- 章节级审校
- 读者体验评分

只有到了这个阶段，才值得认真引入更多专用 Agent。

### 6.6 用户反馈应该如何进入工作流

未来用户反馈不应该再作为 `feedback_x.txt` 存在，而应该作为结构化事件：

- `feedback_type`
  比如风格、节奏、人设、情节方向、措辞

- `feedback_scope`
  计划级 / 章节级 / 场景级 / 局部选区级

- `feedback_action`
  重写、微调、保留结构重写文风、只改结尾、增强主角存在感

- `feedback_text`
  用户自然语言描述

- `feedback_author`
  作者本人 / 协作者 / 编辑 / 未来读者评论映射

这会直接决定未来系统是否能自然支持：

- 多轮修改历史
- 反馈统计
- 训练评测集
- 协作编辑
- 社区评论反哺创作

## 7. Agent 体系演进规划

### 7.1 当前阶段不要追求 Agent 数量，而要做 Agent 分层

长期建议把 Agent 分成三层，而不是平铺一堆角色名。

#### 第一层：核心生产 Agent

负责直接产出小说内容：

- Planner
- Writer
- Revise

#### 第二层：质量与约束 Agent

负责判断和提出修正建议：

- Critic
- Consistency Checker
- Style Evaluator
- Safety Reviewer

#### 第三层：结构化提取 Agent

负责把文本变成系统可用的结构化状态：

- Character Extractor
- Timeline Extractor
- Foreshadow Extractor
- Summary Builder

这第三层非常重要，因为它决定系统能不能真正拥有“长期记忆”。

### 7.2 每个 Agent 必须有固定契约

建议为每个 Agent 定义统一注册信息：

- `agent_key`
- `purpose`
- `supported_workflow_nodes`
- `input_schema`
- `output_schema`
- `prompt_template`
- `model_policy`
- `retry_policy`
- `fallback_policy`

这样做的价值很直接：

- 可以安全替换模型
- 可以比较两个 prompt 版本
- 可以做 shadow run
- 可以在 harness 中独立测一个节点

### 7.3 Agent 输出要尽量结构化

未来应尽量减少“完全自由文本 + 正则解析”的模式。

建议：

- 章节正文仍然是文本
- 评审意见、打分、结构提取、问题列表，全部收敛到 JSON schema

长期稳定性的关键，不是“输出得像人”，而是“系统能可靠读懂它”。

## 8. Harness 规划

### 8.1 Harness 的定位

StoryForge AI 长期最重要的工程资产之一，不是 Prompt 文件，而是 Harness。

Harness 要回答的核心问题是：

> 当我们改了 prompt、模型、检索策略、工作流节点之后，系统是真的更好了，还是只是看起来更忙了？

### 8.2 Harness 必须覆盖三类评测

#### A. 节点级评测

只测试某个节点本身：

- planner 输出结构是否稳定
- critic 问题定位是否具体
- revise 是否真正修复问题
- 提取器是否能稳定抽取人物、时间线、伏笔

#### B. 章节级评测

测试单章完整链路：

- 是否符合大纲
- 是否符合设定
- 是否有足够强的开头钩子和结尾悬念
- 是否控制在合理字数与可读性范围

#### C. 长程一致性评测

这是小说产品和普通写作助手最大的差异点。

需要评测：

- 多章角色一致性
- 设定冲突率
- 伏笔回收率
- 情节推进合理性
- 重复桥段率

### 8.3 Harness 数据集应如何构建

建议建立三个层次的数据集。

#### 第一层：Golden Set

由团队手工精选的一批“代表性创作任务”：

- 都市爽文
- 玄幻升级
- 女频情感
- 轻悬疑
- 短篇反转

每个样本都固定：

- 用户原始 idea
- 目标平台
- 风格要求
- 章节长度要求
- 期望质量重点

这是回归测试的基础。

#### 第二层：Failure Set

专门收集真实失败案例：

- 人设跑偏
- 标题错位
- 字数严重偏差
- 情节跳跃
- 文风 AI 味重
- 忘记用户修改意见

这类样本的价值极高，因为它直接告诉团队该修什么。

#### 第三层：Production Replay Set

从真实线上创作中匿名抽样，形成可重复回放的数据集，用于观察系统版本变更后是否退化。

### 8.4 Harness 指标建议

建议至少覆盖以下指标：

- `outline_adherence`
  是否跟随本章大纲

- `setting_consistency`
  是否违反设定圣经

- `character_consistency`
  角色行为、语气、动机是否稳定

- `hook_strength`
  开头吸引力和结尾悬念强度

- `readability`
  段落、节奏、语言自然度

- `rewrite_effectiveness`
  修订是否真正消除了 critic 问题

- `cost_per_chapter`
  单章成本

- `latency_per_chapter`
  单章耗时

- `human_acceptance_rate`
  用户一次通过率

### 8.5 Harness 的执行机制

建议建立两个回路。

#### 回路一：离线回归

每次关键改动后，跑固定样本集，比较：

- 总分变化
- 每维度变化
- 成本变化
- 耗时变化

#### 回路二：线上观测

收集真实用户侧指标：

- 首章通过率
- 单章平均重写次数
- 用户手动编辑比例
- 放弃率
- 从 idea 到发布的完成率

长期优化必须是这两个回路一起看。

### 8.6 Harness 的组织方式

建议后续在仓库里形成独立目录，例如：

- `evals/datasets/`
- `evals/runs/`
- `evals/scorers/`
- `evals/reports/`

评测报告不要只存在日志里，要形成标准输出。

## 9. 数据层演进规划

### 9.1 长期最重要的改造方向

长期来看，StoryForge AI 的数据层必须完成一件事：

**把“生成结果”升级为“版本化创作资产”。**

### 9.2 建议的数据分层

#### A. 业务主表

保留并扩展当前：

- `users`
- `projects`
- `chapters`
- `generation_tasks`
- `chapter_versions`
- `share_links`
- `reading_progress`

这是产品业务层。

#### B. 创作运行层

新增一组真正支撑 workflow 的表：

- `workflow_definitions`
- `workflow_runs`
- `workflow_step_runs`
- `workflow_events`
- `workflow_checkpoints`

职责分别是：

- 什么流程
- 这次跑了哪个流程
- 每一步跑成什么样
- 过程中发生了什么事件
- 可以从哪里恢复

#### C. 创作工件层

这是最关键的一层。

建议新增：

- `artifacts`
- `artifact_versions`
- `artifact_links`

工件类型可以包括：

- `idea_brief`
- `novel_plan`
- `setting_bible`
- `chapter_outline`
- `chapter_draft`
- `chapter_review`
- `chapter_revision`
- `chapter_score`
- `chapter_summary`
- `timeline_state`
- `character_state`
- `foreshadow_state`
- `publish_payload`

所有创作产物都应可版本化。

#### D. 反馈与协作层

新增：

- `feedback_items`
- `feedback_threads`
- `collaboration_events`

用来承载：

- 作者反馈
- 协作者反馈
- 未来编辑反馈
- 未来读者评论映射为创作建议

#### E. 模型与评测层

新增：

- `model_calls`
- `prompt_versions`
- `eval_runs`
- `eval_samples`
- `eval_scores`

这一层是优化引擎。

### 9.3 数据事实源优先级

建议明确以下优先级：

1. `PostgreSQL`
   业务事实源 + 创作事实源

2. `Object/File Storage`
   工件原文快照、导出文件、调试回放

3. `Vector Index`
   检索索引，可由 artifacts 重建

4. `Cache / Celery Result`
   临时执行状态，不可作为权威来源

### 9.4 为什么不能继续依赖文件作为主状态

文件当然可以继续保留，但角色应该改变：

- 用于本地调试
- 用于导出
- 用于创作快照归档
- 用于灾难恢复的只读证据

而不是继续承担：

- 当前章节状态
- 当前任务恢复位置
- 用户反馈主记录

因为一旦进入多人协作、社区反馈、移动端创作，这种模式会非常快地失控。

### 9.5 检索层未来的正确位置

长期建议把向量检索视为派生层：

- 从 `artifact` 和 `state snapshot` 生成 embedding
- 按项目、分支、章节范围做硬隔离
- 支持重建与回放
- 与业务状态解耦

换句话说：

`检索层应该依赖数据层，而不是和数据层一起承担事实源角色。`

## 10. 建议的阶段性演进路线

### 阶段 A：0-2 个月，稳住当前引擎

目标不是大改，而是把现有系统从“能跑”提升到“可控”。

优先做：

- 收口文档与代码架构描述，统一当前真实主路径
- 把章节确认、重写、恢复逻辑从文件驱动逐步迁到数据库事件
- 把质量分数、评审意见、用户反馈结构化存储
- 清理全局状态依赖，特别是项目上下文与向量检索上下文
- 为核心生成路径补齐回归测试

阶段目标：

- 同一项目的最新状态可以在数据库中完整回答
- 不需要依赖“看看目录里有没有这个文件”来判断流程走到哪

### 阶段 B：2-4 个月，建立工作流运行时

这是整个架构升级的第一里程碑。

优先做：

- 建立 `workflow_runs` / `workflow_step_runs`
- 把 `NovelOrchestrator` 重构为“节点执行器 + 状态推进器”
- 引入显式 checkpoint
- 支持节点重试、跳过、回放
- 将用户审批改成标准 `approval` 节点

阶段目标：

- 章节级生成可以从任意稳定节点恢复
- 能看见一条 run 里每一步到底发生了什么

### 阶段 C：4-8 个月，建立 Harness 与优化闭环

这是 AI 产品真正开始具备工程飞轮的时间点。

优先做：

- 建立 golden set / failure set
- prompt 版本化
- model call 追踪
- 自动离线回归
- 线上质量指标与通过率指标联动

阶段目标：

- 能用证据比较两个版本的工作流与 prompt
- 系统优化从“拍脑袋”进入“有基准、有回归”的状态

### 阶段 D：8-12 个月，引入结构化记忆与场景级创作

优先做：

- 角色卡、时间线、伏笔表结构化提取
- 章节拆场景
- 场景级生成与拼装
- 章节级一致性检查

阶段目标：

- 解决长篇创作最难的问题：多章一致性
- 为未来的“AI 帮你补全细节”建立更细颗粒度的能力

### 阶段 E：12 个月以后，支撑平台化生态

当创作内核稳定后，再向平台能力自然延伸：

- 发布流水线
- 作者主页
- 作品广场
- 评论与追更
- 打赏与分成
- 读者评论反哺创作

这个阶段数据层与 workflow 层已经具备承载这些能力的基础，不需要重写底座。

## 11. 未来 6 个月最值得落地的工程项目

如果只从“长期收益 / 实施难度 / 对当前 demo 的兼容性”综合判断，接下来最值得做的是下面 6 个项目。

### 11.1 项目一：Run / Step / Artifact 三件套

这是第一优先级。

如果没有这三件套，系统仍然会继续依赖隐式状态和文件推断。

### 11.2 项目二：用户反馈结构化

把“用户说这章不行”变成可分析、可追踪、可复用的数据。

### 11.3 项目三：Prompt 与模型版本登记

否则后面你根本无法准确回答：

- 某次生成用了什么版本
- 为什么表现变了
- 是 prompt 影响还是 workflow 影响

### 11.4 项目四：向量检索去全局状态化

这是一个稳定性项目，不够性感，但长期非常关键。

### 11.5 项目五：最小可用 Harness

不需要一开始就上复杂评委团，先把：

- 固定样本
- 固定流程
- 固定报告

跑起来。

### 11.6 项目六：章节分支与版本回滚

这会显著提升创作者体验，也会直接增强平台的“创作工作台”属性。

---

## 12. 明确的反模式

长期项目里，有几件事现在看起来很诱人，但应该明确避免。

### 12.1 反模式一：先堆 Agent 数量

如果 workflow、contract、harness 还没建立，Agent 越多，系统越脆。

### 12.2 反模式二：先做复杂微服务化

当前更需要的是边界清晰，而不是进程数量变多。

### 12.3 反模式三：把向量库当主数据库

向量库是检索层，不是创作真相。

### 12.4 反模式四：只追求生成效果，不追求可恢复性

长篇创作不是一次性回答，恢复能力和版本能力与生成质量同样重要。

### 12.5 反模式五：没有 Harness 就频繁改 Prompt

这会让团队越来越依赖直觉，越来越难积累系统性能力。

---

## 13. 一句话总结

StoryForge AI 的长期架构方向，不应该是“把今天的 orchestrator 改得更长”，而应该是：

**从一个线性生成脚本，演进成一个以工作流、工件、评测和数据资产为中心的 AI 原生创作系统。**

如果这个方向走对了，未来无论你要做：

- 更强的多智能体写作
- 更稳定的人机协作
- 作者工作台
- 小说广场
- 评论反哺创作
- 打赏与商业化

底层都会是顺着长出来，而不是每走一步都要推倒重来。
