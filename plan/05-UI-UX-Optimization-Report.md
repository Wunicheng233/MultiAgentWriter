# UI/UX 优化报告：StoryForge 前端体验升级

> **问题诊断**：当前UI信息密度过高，视觉层次混乱，用户难以找到重点，无法沉浸在创作心流中。

---

## 一、现状诊断：核心问题

### 1.1 ProjectOverview 页面：信息过载的指挥中心

**问题定位**：`frontend/src/pages/ProjectOverview.tsx`（1000+ 行代码）

**问题清单**：

| 问题 | 严重程度 | 具体表现 |
|------|---------|---------|
| **一屏信息过载** | 🔴 高 | 单屏展示了9个功能模块：项目状态、工作流、4个Agent、配置、质量、交付、产物、协作者 |
| **视觉层次扁平** | 🟠 中 | 所有卡片使用相同边框、相同阴影、相同视觉重量，用户无法快速识别优先级 |
| **CTA 按钮爆炸** | 🔴 高 | 顶部状态卡有4个按钮，每个功能区又有2-3个按钮，共约12个操作入口 |
| **渐进式披露缺失** | 🟠 中 | 所有信息一次性展示，没有折叠/展开、标签切换等信息管理机制 |
| **危险操作暴露** | 🟡 低 | "重置项目"按钮在页面底部最显眼位置，无确认保护 |

**页面模块分布图**：

### 1.2 Editor 页面：写作空间被挤压

**问题定位**：`frontend/src/pages/Editor.tsx`（600+ 行代码）

**问题清单**：

| 问题 | 严重程度 | 具体表现 |
|------|---------|---------|
| **左侧 Agent 面板占用空间** | 🟠 中 | 320px 固定宽度，写作时不需要一直看4个Agent状态 |
| **状态栏信息冗余** | 🟡 低 | 顶部状态卡展示8个信息点，大部分写作时不需要 |

### 1.3 系统性问题

1. **缺少模式切换**：用户在不同阶段（策划、浏览、写作、评审）需要的信息完全不同，但UI是统一的
2. **状态展示优先级缺失**：所有状态信息同等重要地展示，没有按上下文过滤
3. **动画和过渡效果不足**：页面切换、状态变化时缺少平滑过渡，认知负担重
4. **移动端体验**：响应式布局只做到了"能看"，没有针对移动端优化交互

## 二、优化原则

### 2.1 核心理念：上下文感知的渐进式披露

```
"在正确的时间，只展示正确的信息"
```

### 2.2 设计原则

| 原则 | 说明 |
|------|------|
| **优先级分层** | 信息按"当前必须"、"偶尔需要"、"背景参考"分层展示 |
| **模式切换** | 创作模式 ≠ 监控模式 ≠ 配置模式，不同模式不同UI |
| **渐进式披露** | 默认隐藏次要信息，需要时才展开 |
| **留白呼吸** | 增加负空间，降低视觉噪音 |
| **心流保护** | 关键操作（写作、确认）不被打断 |

## 三、具体优化方案

### 3.1 ProjectOverview：从"仪表盘"到"指挥中心"

**目标**：将信息密度降低60%，核心行动路径清晰可见。

#### 方案1：三标签页架构

```
┌──────────────────────────────────────────────────────────┐
│  [ 返回书架 ]  项目标题  [状态Badge]                      │
├──────────────────────────────────────────────────────────┤
│  ┌──────────┐ ┌──────────┐ ┌──────────┐                  │
│  │  工作流  │ │  创作配置 │ │  质量交付 │  ← 三个标签页  │
│  │  (默认)  │ │          │ │          │                  │
│  └──────────┘ └──────────┘ └──────────┘                  │
├──────────────────────────────────────────────────────────┤
│                                                           │
│  当前标签页内容（每次只显示一个）                          │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

**标签页1：工作流** (默认)
- 顶部：当前状态卡（简化版，只显示关键信息 + 主CTA）
- 中间：5步工作流时间线（当前步骤高亮，其他步骤折叠）
- 下方：4个Agent状态条（水平排列，不是卡片）

**标签页2：创作配置**
- 项目配置、Core Requirement、模式设置
- 协作者管理移到这里

**标签页3：质量与交付**
- 质量评分、进度统计
- 导出功能、分享链接
- Artifacts列表

#### 方案2：状态卡片简化

**当前**：80%的信息，20%的行动
**优化后**：20%的信息，80%的行动

```
优化后的状态卡片：
┌──────────────────────────────────────────────────────────┐
│  [状态图标]  第 3 章正在等待人工确认                       │
│  当前章节已通过 Guardrails 检查，等待你的审阅后继续生成     │
│                                                           │
│  [ 进入写作台 → ]    [ 查看章节列表 ]    [ 质量分析 ]      │
└──────────────────────────────────────────────────────────┘
```

#### 方案3：危险操作保护

- "重置项目"移到设置页面的最底部
- 增加二次确认（输入项目名才能确认）
- 视觉上用灰色弱化，不突出显示

### 3.2 Editor：从"工作台"到"写作空间"

**目标**：将写作空间占比从70%提升到90%，提供无干扰写作体验。

#### 方案1：专注写作模式（默认）

```
┌──────────────────────────────────────────────────────────┐
│  [ ≡ ]  第 3 章                              [ ⟳ ] [ ✓ ]  │
│  汉堡菜单                                         保存    │
├──────────────────────────────────────────────────────────┤
│                                                           │
│                                                           │
│                    写作画布（全宽）                        │
│                                                           │
│                                                           │
│                                                           │
├──────────────────────────────────────────────────────────┤
│  2,456 字  ·  自动保存: 已就绪  ·  Agent: 空闲            │
└──────────────────────────────────────────────────────────┘
```

**特性**：
- 左侧 Agent 面板默认收起，通过汉堡菜单展开
- 顶部状态信息简化到一行
- 只有必要的操作按钮（重新生成、保存）
- 底部状态栏展示关键数据

#### 方案2：非模态确认系统

**问题**：当前全屏模态框确认完全打断写作心流
**优化**：右侧滑出确认面板

```
┌──────────────────────────────────────────┬─────────────────┐
│                                          │  [ 确认面板 ]   │
│                                          │                 │
│                                          │  第3章已生成    │
│      写作画布                            │                 │
│                                          │  [ 预览 ]       │
│                                          │                 │
│                                          │  [ 修改意见 ]   │
│                                          │                 │
│                                          │  [ 通过 ] [ 不通过 ] │
└──────────────────────────────────────────┴─────────────────┘
```

**优点**：
- 不覆盖写作内容，用户可以对照查看
- 非全屏遮罩，认知负担轻
- 可以随时关闭，不强制用户立即决策

#### 方案3：Agent状态按需展示

- 写作时：Agent状态只在底部状态栏显示一行文字
- 生成时：自动展开左侧面板，展示实时进度
- 完成后：自动收起，恢复专注模式

---

### 3.3 系统性优化

#### 视觉层次重构

| 元素类型 | 当前 | 优化后 |
|----------|------|--------|
| **主CTA** | Sage绿色，全圆角 | 增加微阴影，hover状态有微小放大 |
| **次要按钮** | 边框灰色 | 降低对比度，减少视觉重量 |
| **状态卡片** | 渐变背景 + 边框 | 只有边框，无渐变，降低视觉噪音 |
| **信息卡** | 全边框 | 无边框，只用间距分组 |
| **进度条** | 全宽 | 更细，背景色更淡 |

#### 动画与过渡

- 页面切换：0.3s 淡入淡出 + 微小Y轴位移
- 面板展开/收起：0.2s 宽度过渡 + 内容淡入
- 状态变化：0.15s 颜色过渡，不闪烁

## 七、Codex 独立复核与修订后的执行计划

### 7.1 我同意的判断

- **ProjectOverview 的主要问题确实是信息同屏堆叠**。但它不是单纯“卡片太多”，而是缺少任务语境：用户当前是在监控工作流、改配置，还是准备交付。
- **Editor 的核心问题确实是写作画布被过程信息挤压**。工作流过程很重要，但它应该按需出现，而不是永久占据 320px。
- **危险操作需要降权和确认保护**。重置项目不能作为页面末尾的普通文本按钮裸露。

### 7.2 我调整的部分

- 不把 ProjectOverview 一次性改成全新的大重构。为了降低回归风险，本轮采用**轻量标签页**承载三种模式：`工作流`、`创作配置`、`质量交付`。
- 不隐藏 workflow v2 过程。我们的产品卖点之一就是“可审计的智能体创作过程”，所以不是把过程弱化，而是把它放进正确的上下文里。
- Editor 的确认流程先改成**右侧滑出面板**，保留原有确认 API 和数据流，不引入新的状态机。

### 7.3 本轮已落地范围

- [x] ProjectOverview 顶部状态卡降噪：保留项目状态、当前焦点、主 CTA、章节入口和当前 Run 入口。
- [x] ProjectOverview 增加三标签页：工作流 / 创作配置 / 质量交付。
- [x] Agent 展示从 4 张同权重大卡片改为水平状态条，减少视觉重量。
- [x] 创作配置、协作者和项目维护归入 `创作配置` 标签。
- [x] 质量评分、导出分享和 artifacts 归入 `质量交付` 标签。
- [x] 重置项目改为维护区弱化入口，并要求输入完整项目名后才能确认。
- [x] Editor 默认扩大写作画布，过程面板默认收起。
- [x] Editor 生成中或等待确认时自动展开过程面板。
- [x] Editor 人工确认从全屏遮罩模态改为右侧滑出面板。
- [x] 按钮和进度条视觉层级统一收敛到 `00-DESIGN.md` 的纸感、Sage Green 和低噪音原则。

### 7.4 下一轮建议

- 将版本历史从侧栏进一步改成底部抽屉，避免和 Agent 过程争夺同一块空间。
- WorkflowRunDetail 可以增加事件分组和折叠，避免长 run 形成日志墙。
- 移动端需要单独做一次触屏体验检查：标签页、确认面板、写作画布高度和底部操作栏都要按真实手机宽度验证。
- 加一组页面级可视化回归截图，至少覆盖 ProjectOverview 三个标签和 Editor 专注/过程/确认三种状态。
