# StoryForge AI 两周比赛冲刺计划

时间窗口：

- 2026-04-23 至 2026-05-06

---

## 1. 文档定位

这份文档不是替代长期路线图，而是在比赛周期被压缩到两周之后，为团队提供一份真正可执行的短期冲刺计划。

它解决三个问题：

1. 接下来两周我们到底要交付什么
2. 当前项目真实进度到哪里了
3. 如何在不牺牲长期潜力的前提下，把产品打磨到“可展示、可试用、可讲故事”的完成度

因此，从今天开始，项目采用双轨策略：

- `比赛冲刺轨`
  目标是在 2026-05-06 前交付一个完成度高、主路径稳定、叙事清晰的比赛版本
- `赛后延伸轨`
  不推翻现有长期规划，继续沿 `workflow / artifact / harness / workspace` 主线演进

---

## 2. 当前现状总结

基于当前仓库代码，而不是只看历史规划文档，StoryForge AI 已经不是一个只有概念的 demo，而是一个：

`具备前后端工作台、异步生成链路、多角色 Agent 流程、章节编辑、质量面板、导出与分享能力的 AI 写作产品原型`

### 2.1 已经完成到什么程度

#### A. 产品层能力已经具备雏形

当前前端已经具备以下页面和路径：

- 登录 / 注册
- 项目列表 Dashboard
- 多步骤创建项目页
- 项目概览页
- 章节列表页
- 编辑器 / 阅读器
- 质量分析页
- 设置页
- 分享阅读页

这说明产品主界面骨架已经形成，不是从零开始。

#### B. 多 Agent 创作主链路已经可运行

当前后端和核心编排层已经具备：

- `Planner -> Writer -> Guardrails -> Critic -> Revise` 的主链路
- 人工确认与继续生成机制
- Celery 异步任务执行
- 项目/章节/任务持久化
- 世界观与向量检索支持
- 章节评测结果标准化

从代码现实看，当前系统的“多智能体”不是空口概念，而是已经落在编排和数据模型里的。

#### C. 长期扩展底座已经开始出现

数据库中已经引入：

- `WorkflowRun`
- `WorkflowStepRun`
- `Artifact`
- `FeedbackItem`
- `ChapterVersion`
- `ShareLink`
- `ProjectCollaborator`
- `TokenUsage`

这意味着系统已经开始从“脚本式生成器”向“可追踪工作流系统”过渡，这一点是后续扩展空间最宝贵的资产。

#### D. 比赛可展示能力已经有基础

目前已经具备适合比赛演示的几个亮点：

- 多 Agent 协同创作叙事
- 人在环路确认
- 质量分析看板
- 导出 `EPUB / DOCX / HTML`
- 公开分享链接
- 协作者机制

这套能力组合已经足以支撑一次有说服力的产品演示。

### 2.2 当前真实短板

#### A. 比赛版本最大短板不是“没功能”，而是“收口不够”

现在最危险的问题，不是功能数量太少，而是：

- 主路径虽然完整，但稳定性和异常恢复还不够令人放心
- 文档里的长期愿景远大于两周内真正可交付的部分
- 评委第一眼能感知到的产品体验，还没有被刻意收束成一个高完成度故事

#### B. 当前系统仍然偏“工程原型”，不是“精致产品”

主要表现为：

- 创建项目流程仍然比较表单化
- 核心亮点分散在不同页面，价值没有被主动组织出来
- 用户不一定能立刻理解“为什么一定要多 Agent”
- 比赛评委更在意的“完成一部作品的确定性”还需要更强展示

#### C. 架构层仍有明显技术债

当前仍存在这些结构性问题：

- 数据库、文件系统、Celery 状态并存，事实源没有完全收口
- 核心编排仍以串行 orchestrator 为主
- workflow / artifact / feedback 虽已落库，但还没有完全成为唯一事实源
- 测试存在，但当前环境未安装 `pytest`，无法在此工作区直接完成自动化回归验证

### 2.3 现阶段进度判断

如果用比赛视角来判断当前项目，不建议再说“还在早期 demo”，那会低估你们已经拥有的积累。

更准确的说法是：

- `创作工作台基础能力`：约 `70%`
- `比赛版可演示完成度`：约 `55%`
- `长期平台化能力`：约 `25%`

这组数字不是精确测量，而是用于帮助团队统一判断：

- 你们不是从 0 到 1
- 你们现在是在做 `0.6 -> 0.9` 的比赛冲刺
- 真正要放弃的是“几个月版本的贪心范围”，不是放弃项目野心

## 3. 路线改法：三种冲刺策略

### 方案 A：纯 Demo 打磨

做法：

- 只修 UI、演示文案和少量 bug
- 不碰工作流底座

优点：

- 短期见效快

缺点：

- 一旦现场试用稍微超出预设脚本，容易暴露稳定性问题
- 赛后需要重新回头补底座

### 方案 B：主路径收口 + 亮点包装 + 扩展留缝

做法：

- 只围绕比赛主路径做稳定性、体验和叙事收口
- 同时保留 workflow / artifact / harness 的扩展方向

优点：

- 最符合两周时间现实
- 既能提升比赛表现，也不会把赛后路线带偏

缺点：

- 必须严格控制范围，不能临时加入太多新功能

### 方案 C：继续按长期架构推进

做法：

- 继续做 workflow runtime、上下文去全局化、harness 深化等长期工程

优点：

- 长期正确

缺点：

- 两周内用户可感知成果不足
- 容易输在“产品完成度”和“讲故事能力”

### 推荐方案

本阶段明确选择 `方案 B`。

也就是：

**两周内不追求把长期架构做完，而是把“从创意到可分享作品”的比赛主路径打磨到足够强，同时所有新增改动都尽量挂靠到已有的 workflow / artifact / evaluation 基础上。**

## 4. 两周冲刺目标

比赛版本只定义一个核心目标：

**让评委在一次 5 到 10 分钟的体验里，清楚看到 StoryForge AI 可以把一个小说创意，稳定推进成一部可编辑、可评审、可导出、可分享的作品。**

围绕这个目标，比赛版必须做到四件事：

1. 主路径稳定
2. 产品亮点集中
3. 演示过程流畅
4. 赛后继续扩展时不用推倒重来

---

## 5. 比赛版范围定义

### 5.1 必做范围

#### P0. 创作主路径稳定化

必须确保下面这条路径可稳定演示：

`创建项目 -> 生成策划 -> 逐章写作 -> 评审修订 -> 查看质量 -> 编辑章节 -> 导出 / 分享`

两周内最优先修的，不是新功能，而是这条链上的所有中断点：

- 任务状态不一致
- 等待确认后继续生成不稳定
- 章节/任务/工作流状态展示不清楚
- 导出和分享的最后一公里问题

#### P0. 演示脚本产品化

需要把当前零散能力收束成清晰的产品故事：

- 为什么是多 Agent，而不是一次性长回答
- 用户在哪些节点拥有控制权
- 为什么系统能比纯聊天更适合长篇创作
- 输出结果如何进入“可继续创作”和“可发布分享”状态

这部分不只是讲稿问题，也应反映到产品界面和页面顺序里。

#### P1. 关键页面完成度提升

优先打磨这些页面：

- 创建项目页
- 项目概览页
- 编辑器 / 阅读器
- 质量分析页
- 分享阅读页

目标不是全面重做，而是提升：

- 信息层级
- 可理解性
- 评委第一次上手时的顺滑程度

#### P1. Demo 数据与模板准备

比赛版本需要预置高质量演示素材：

- 2 到 3 个有吸引力的创作题材模板
- 一条稳定可复现的演示流程
- 至少一个完整作品样例

这样比赛时不依赖临场 prompt 发挥。

### 5.2 可以做，但不能挤压主路径

#### P2. 工作流可视化增强

如果时间允许，可以增强：

- 当前 run 的步骤展示
- 章节级评审报告摘要
- revision 前后差异感知

这会显著增强“多 Agent 真在协作”的说服力。

#### P2. 成本与效率可观测

可以适度补强：

- token 消耗可视化
- 单章完成时间
- 任务阶段耗时

这会帮助比赛答辩时解释产品工程能力。

### 5.3 这两周明确不做

- 不做社区广场
- 不做推荐流
- 不做创作者商业化
- 不做大量新增 Agent 角色
- 不做复杂 DAG 工作流 UI
- 不做大规模重构到“纯数据库事实源”
- 不做需要数周验证的新型风格系统

## 6. 两周执行计划

### 第 1 周：把主路径收口

时间：

- 2026-04-23 至 2026-04-29

目标：

- 让比赛主路径稳定、可解释、可复现

本周重点：

1. 梳理并修复主路径上的高频失败点
2. 统一任务状态、等待确认、继续生成、重试的交互表现
3. 补足概览页和质量页上对“当前生成到哪一步”的说明
4. 校验导出、分享、协作这些比赛可感知能力
5. 准备稳定的 demo 项目和演示脚本

周末验收标准：

- 团队内部可以连续演示一遍完整流程
- 没有明显卡死、跳状态、结果丢失问题
- 每个页面都能说清楚它在整体故事中的作用

### 第 2 周：把产品完成度做出来

时间：

- 2026-04-30 至 2026-05-06

目标：

- 让产品不只是“能跑”，而是“像一个完成度高的参赛作品”

本周重点：

1. 优化关键页面视觉和文案，使亮点更集中
2. 强化多 Agent 协作、质量闭环、人在环路的展示感
3. 补演示兜底方案：
   - 网络不稳怎么办
   - 生成耗时过长怎么办
   - 现场需要快速展示完整结果怎么办
4. 完成比赛材料配套：
   - 演示脚本
   - 样例项目
   - 一句话价值主张
   - 关键对比点

最终验收标准：

- 陌生人第一次看演示能在几分钟内理解产品价值
- 核心页面不会让人误以为这只是“包装成网页的聊天机器人”
- 有完整样例支撑评委快速看到结果

## 7. 建议的任务分组

为了两周内提高速度，建议按四条线并行，而不是所有人同时盯着一个大列表。

### A 线：主路径与后端稳定性

负责人关注：

- 任务状态流转
- workflow / task / chapter 一致性
- 等待确认与继续生成
- 导出与分享兜底

目标：

- 确保比赛现场不会被链路故障打断

### B 线：前端完成度与信息组织

负责人关注：

- 创建流程
- 概览页
- 质量页
- 分享页
- “多 Agent 协作”可理解性

目标：

- 让评委第一次上手就能明白价值

### C 线：Prompt / 评审 / Demo 内容

负责人关注：

- 高质量演示样例
- 题材模板
- 章节质量稳定性
- 批评和修订结果是否足够有说服力

目标：

- 让 AI 产出在比赛中显得有质量、有过程、有改进

### D 线：答辩与产品叙事

负责人关注：

- 一句话介绍
- 核心差异点
- 演示路径
- 现场提问的答案准备

目标：

- 把工程成果变成评委能记住的产品叙事

## 8. 比赛版成功标准

如果两周后只能用最少的标准判断成败，我建议看下面五条：

1. 能否稳定完成一条完整创作主路径演示
2. 评委能否理解“多 Agent + 人在环路 + 质量闭环”的价值
3. 是否有一部可展示的完整样例作品
4. 导出与分享是否真的可用，而不是纸面功能
5. 比赛后继续开发时，是否仍能沿当前 workflow / artifact / harness 主线前进

只要这五条成立，这次冲刺就是成功的。

---

## 9. 赛后扩展空间保留策略

两周冲刺不是否定长期方向，而是把长期方向分层处理。

比赛结束后，优先继续沿下面顺序扩展：

1. `Workflow Runtime`
   让 `WorkflowRun / WorkflowStepRun / Artifact` 进一步成为真正的一等运行时对象
2. `Evaluation Harness`
   从当前章节评审，走向更可回归、可量化的质量验证体系
3. `Author Workspace`
   把编辑器、设定、反馈、版本和局部重写能力做成真正长期使用的创作工作台
4. `Style Runtime`
   在合规和治理前提下做风格控制、风格档案、风格对比
5. `Publishing / Community`
   再去扩展读者反馈、连载发布、作者主页和社区机制

也就是说：

**比赛版追求“把一条链路做到漂亮”，赛后版本追求“把整个平台做深”。**

---

## 10. 结论

StoryForge AI 当前最需要的，不是推翻原计划，而是把原计划拆成：

- 现在必须赢下的两周
- 之后继续做大的半年到一年

对当前状态的最好判断不是“做不完了”，而是：

**底座已经有了，问题在于范围太大。**

所以从 2026-04-23 起，团队统一目标应调整为：

**在 2026-05-06 前，交付一个主路径稳定、亮点集中、可完整演示的多智能体写作产品；比赛后继续沿现有架构主线扩展，而不是另起炉灶。**
